# 异常

# 异常概述

异常是指在 Java 程序运行过程中可能出现的一些错误，它可能由程序员没有预料到的各种情况产生，也可能由超出了程序员可控范围的环境因素产生。
Throwable 类是所有错误（Error）和异常（Exception）的父类，只有 Throwable 类型的实例才可以抛出（throw）或者捕获（catch）。

  • Error

    • Error 表示严重的错误，绝大部分 Error 会导致程序处于非正常的、不可恢复的状态。

    • 编译器不会检查 Error 。

    • 常见错误有：

      错误说明
      AssertionError断言错误
      VirtualMachineError虚拟机错误
      UnsupportedClassVersionErrorJava 类版本错误
      StackOverflowError栈溢出错误
      OutOfMemoryError内存溢出错误

  • Exception

    • Exception 是程序运行中出现的意外情况，可以被捕获，并进行相应处理。

    • 编译器会检查可检查的 Exception ，如果没有进行异常处理，则不能通过编译。

    • 常见异常有：

      异常说明
      ClassNotFoundException应用程序加载类时，找不到相应的类，抛出该异常
      CloneNotSupportedException调用 clone 方法克隆对象，但该对象的类没有实现 Cloneable 接口时，抛出该异常
      IllegalAccessException拒绝访问一个类的时候，抛出该异常
      InstantiationException当使用 Class 类中的 newInstance 方法创建一个类的实例，而指定的类对象无法实例化，抛出该异常
      NoSuchFieldException请求的变量不存在
      NoSuchMethodException请求的方法不存在
      SocketException读取网络失败

  • RuntimeException

    • RuntimeException 表示运行时异常，是 Exception 的一个子类。

    • 编译器不会检查运行时异常，可能导致运行时异常的语句没有异常处理也可以通过编译。

    • 常见编译时异常：

      运行时异常说明
      NullPointerEXception当对 null 调用方法或字段时，抛出该异常
      IndexOutOfBoundsException当对某排序索引值超出范围时抛出该异常
      StringIndexOutOfBoundsException当对字符串使用错误索引值时，抛出该异常
      ClassCastException当强制将对象转换为不是实例的子类时，抛出该异常

Java 规定：

  • 必须捕获的异常：包括 Exception 及其所有非 RuntimeException 子类，这类异常被称为可检查的异常（Checked Exception）。
  • 不需要捕获的异常：包括 Error 及其子类， RuntimeException 及其子类。

# 异常处理

# 抛出异常

throw 和 throws 用于抛出异常。
throw 放在方法体中，后面抛出异常对象。
throws 放在方法签名的尾部，后面接异常类型，多个异常类型可以用 , 隔开。
在方法重写中，如果父类的方法抛出了异常，子类重写的方法可以不用抛出异常；但是如果父类方法没有抛出异常，则子类一定也不能抛出异常。

抛出的错误会到调用它的方法中处理，如果没有被处理，则会继续抛给这个调用方法的上层方法，这个过程会一直持续到异常被处理或者到达 main() 方法。
如果 main() 方法中也没有处理异常，则 JVM 会将异常信息打印到控制台上并终止程序运行。
Java 中使用 try...catch...finally 捕获并处理异常。

  • getMessage() ：获取异常的描述信息，原因。
  • printStackTrace() ：打印异常的跟踪栈信息并输出到控制台上。

# 使用说明

如果程序代码中，涉及到资源的调用（流、数据库连接、网络连接等），必须考虑 try..catch...finally 处理，保证不出现内存泄露；
如果程序各方法间是递进调用关系，可以考虑使用 throws ，然后在最上层的方法中使用 try...catch...finally 处理。

# 最佳实践

  • 对可恢复的情况使用检查性异常（ Exception ），对编程错误使用运行时异常（ RuntimeException ），并且尽可能使用 Java 标准的异常。
  • 尽量减少 try 代码块的大小，并且尽可能 catch 较小的异常范围。
  • 尽量不要在 finally 抛出异常或者返回值，因为 finally 中的 throw 和 return 会覆盖 catch 中的语句。
  • 不要忽略异常，如果捕获了异常，就应该处理，即使只是将异常记录到日志文件中，也不要直接丢弃。
  • 各类异常应该进行分类管理，如可分为：逻辑异常（用户制造的意外）、代码错误（程序编写导致的 BUG）和专有异常（特定业务常景的异常）。

异常设计的初衷是解决程序运行中的各种意外情况，且异常的处理效率比条件判断方式要低很多。

不要用异常进行业务逻辑处理。在业务处理的时候，如果发现无法处理直接抛出异常即可，而不是在逻辑处理的时候，用异常来判断逻辑进行的状况。

# 多线程

Java 内置了多线程支持，多线程模型是 Java 程序的最基本一种迸发模型。

# 基础概念

# 线程与进程

  • 程序（program）：为完成特定任务，用某种语言编写的一组指令的集合。
  • 软件（software）：一个或多个应用程序和相关的资源文件构成一个软件系统。
  • 进程（process）：程序的一次执行过程，是对整个过程（创建 - 运行 - 消亡）的描述。系统会为每个运行的程序建立一个进程，并为进程分配独立的系统资源。
  • 线程（thread）：程序内部的一条执行路径，是进程中的一个执行单元，一个进程中至少有一个线程。一个进程中并行多个线程，这个程序可以称为多线程程序。

    多线程可以充分利用 CPU 空闲时间片，用尽可能短的时间完成用户的请求，使程序的响应速度更快 。

# 并行与并发

  • 并行（parallel）：指多个事件在同一时刻发生（同时发生）。指在同一时刻，有多条指令在多个 CPU 上同时执行。单核 CPU 同一时刻只能处理一条指令，做不到并行处理。

  • 并发（concurrency）：指多个事件在同一个微小的时间段内发生。同一个时刻虽然只能有一条指令执行，但多个进程的指令被快速轮换执行，使得在宏观上具有多个进程同时执行的效果。程序并发执行可以在有限条件下，充分利用 CPU 资源。

    单核处理器不能并行处理多个任务，只能是多个任务的并发运行。同理，线程也是一样的。从宏观角度上理解线程是并行运行的，但是从微观角度上分析却是串行运行的，即一个线程一个线程的去运行，当系统只有一个 CPU 时，线程会以某种顺序执行多个线程，这种情况被称为线程调度。

# 线程调度

CPU 资源分配给不同线程的方式，常见的线程调度方式有两种：

  1. 分时调度
    分时调度模型是指让所有的线程轮流获得 CPU 的使用权，平均分配每个线程占用 CPU 的时间片

  2. 抢占式调度
    优先让可运行池中优先级高的线程占用 CPU，如果可运行池中的线程优先级相同，那么就随机选择一个线程，使其占用 CPU。
    处于运行状态的线程会一直运行，直至它不得不放弃 CPU。
    Java 采用的是抢占式调度方式。

    Java 的一个线程可能会因为以下原因而放弃 CPU。

    1. JVM 让当前线程暂时放弃 CPU，转到就绪状态
    2. 当前线程因为某些原因而进入阻塞状态
    3. 线程结束运行

# 创建线程

Java 中有三种方式可以创建线程：

  1. 继承 Thread 类
  2. 实现 Runnable 接口
  3. 实现 Callable 接口

# 继承 Thread

Thread 类实现方式中，每个线程都是通过某个 Thread 对象的 run() 方法来完成操作的。

  • run() 方法被 Thread 的子类重写，其方法体中代表线程需要完成的任务，因此又称其为线程执行体。
  • run() 方法不是直接调用，而是通过子类对象的 start() 方法来启动，想实现多线程，需要创建多个线程对象。

run() 方法通过 start() 方法由 JVM 调用，具体过程控制由于操作系统的 CPU 调度实现。
要想启动多线程必须调用 start() 方法，一个线程对象只能调用一次 start() 方法，否则会抛出 IllegalThreadStateException 异常。

Thread 中的常用方法如下：

  • 构造方法： Thread 的构造方法通过重载可以接受两个参数， Runnable target 和 String name 。
    • target 用于指定创建线程的目标对象，它实现了 Runnable 接口中的 run() 方法。
    • name 是一个字符串，用于指定线程对象的名字。可以调用对应的 setter/getter 方法设置。
  • public void start() ：线程开始执行，JVM 会自动调用线程的 run() 方法。 run() 方法不应该手动调用，如果手动调用则只是普通的方法，没有开启多线程。
  • public static Thread currentThread() ：返回对当前正在执行的线程对象的引用。在 Thread 子类中就是 this ，通常用于主线程和 Runnable 实现类。
  • public static void sleep(long millis) ：使当前正在执行的线程以指定的毫秒数暂停（暂时停止执行）。
  • public static void yield() ：让当前线程暂停一下，使系统的线程调度器重新调度一次，希望优先级与当前线程相同或更高的其他线程能够获得执行机会。
  • public final boolean isAlive() ：测试线程是否处于活动状态。如果线程已经启动且尚未终止，则为活动状态。
  • 优先级相关方法：Java 为抢占式调度，优先级高的线程会获得较多的执行机会。每个线程的默认优先级和创建它的父类相同，同时也可以进行指定：
    • 优先级常量：
      MAX_PRIORITY ：10，最高优先级
      MIN_PRIORITY ：1，最低优先级
      NORM_PRIORITY ：5，普通优先级，默认
    • public final int getPriority() ：返回线程优先级
    • public final void setPriority(int newPriority) ：设置线程优先级，范围 1~10

Java 的类只能单继承，因此继承 Thread 类启动线程的方式存在一定限制。

# 线程通信

当使用多个线程来共同完成一件任务时，为了确保任务按照预期进行，就需要线程间的通信机制来协调多线程间的工作。
一种线程协调机制被称为等待唤醒机制。当一个线程满足某个条件时，使其进入等待状态，当其他线程执行完成指定任务后再将其唤醒。Java 线程中的 wait/notify 方法可以实现这种协作机制。

  • wait() ：线程不再活动，也不再竞争锁对象，如果它之前拥有锁，则会释放锁对象让其他线程竞争。 wait 可以传入等待时间，这样它将在等待时间结束后由系统唤醒，重新开始活动。又或者不设置时间，这样除非其他线程唤醒它，否则这个线程不会进入活动。
  • notify() ：随机选择一个进入等待中的线程进行唤醒， notifyAll() 方法将会唤醒所有线程。

wait() 和 notify() 方法必须在同步代码块或同步方法中使用，而且必须由同一个锁对象调用，它们都是属于 Object 类的方法。

# 线程状态

java.lang.Thread.State 中定义了 6 种不同的线程状态，在给定的一个时刻，线程只能处于其中的一个状态。

  • 新建（New）：
    创建的线程尚未启动。创建后还未调用 start() 方法的线程处于此状态。
  • 就绪（Runnable）：
    线程已经在 JVM 中运行。线程已经调用了 start() 方法，但在操作系统层面，它可能处于运行状态，也可能在等待资源调度，调度完成即可进入运行状态。此状态表明线程可以被运行。
  • 阻塞（Blocked）：
    线程处于阻塞状态，表明线程在等待 synchronized 的隐式锁。当占用锁的线程释放锁，而阻塞中的线程获得锁后，它又转换到就绪状态。
  • 等待（Waiting）：
    线程处于无限期等待，直到被其他线程显式唤醒。与阻塞不同，等待是主动进入的，可以通过调用 Object.wait() 方法等进入。
  • 定时等待（Timed Waiting）：
    和等待状态的区别是，线程无需其他线程唤醒，而是在等待时间满后被系统自动唤醒。 Thread.sleep() 和 Object.wait() 等方法可以设置等待时间参数进入此状态。
  • 终止（Terminated）：
    线程执行完 run() 方法，或因为异常终止运行。此状态意味着线程已经结束了生命周期。

# 可重入锁和死锁

可重入锁，指的是线程可以重复获取同一把锁。即同一个线程在外层方法获取了锁，在进入内层方法会自动获取锁。
可重入锁可以在一定程度上避免死锁。

死锁是一种特定的程序状态。当两个以上的运算单元，双方都在等待对方停止执行，以获取系统资源，但是没有一方提前退出时，就称为死锁。
诱发死锁有四个条件：互斥条件、占用且等待、不可抢占和循环等待。

# 公平锁和非公平锁

  • 公平锁：是指多线程按照申请锁的顺序来获取锁。
  • 非公平锁：是指多线程不按照申请锁的顺序来获取锁 。这就可能会出现优先级反转（后来者居上）或者饥饿现象（某线程总是抢不过别的线程，导致始终无法执行）。

公平锁为了保证线程申请顺序，需要付出一定的性能代价，因此其吞吐量一般低于非公平锁

# 独享锁和共享锁

  • 独享锁是指锁一次只能被一个线程所持有。
  • 共享锁是指锁可被多个线程所持有。

独享锁与共享锁是一种广义上的说法，从实际用途上来看，也常被称为互斥锁与读写锁。

# 乐观锁和悲观锁

乐观锁和悲观锁是指两种不同的处理并发同步的策略。

  • 悲观锁：
    对并发采取悲观的态度。悲观锁认为，不加锁的并发操作一定会出现问题。悲观锁适合写操作频繁的场景。
  • 乐观锁：
    对并发采取乐观的态度。乐观锁认为，不加锁的操作也不一定会发生问题。乐观锁在更新数据时，会采取不断尝试更新的方式，乐观锁适合读多写少的场景。

Java 中使用 synchronized 关键字来显示加锁进行互斥同步，是一种典型的悲观锁。
Java 中的典型乐观锁是 AtomicInteger 等原子类，它们在更新数据时使用了 CAS（Compare And Swap）机制。乐观锁在进行操作时，先进行比较，如果内存位置的值和预期的原值相等，那么认为此时可以安全地把该值更改为新值。

# 偏向锁、轻量级锁和重量级锁

轻量级锁与重量级锁，指的是锁控制粒度的粗细。控制粒度越细，阻塞开销越小，并发性也就越高。

  • 偏向锁：
    如果一段同步代码一直被一个线程所访问，那么该线程会自动获取锁。降低获取锁的代价。

  • 轻量级锁：
    当锁是偏向锁的时候，被另一个线程所访问，偏向锁就会升级为轻量级锁，其他线程会通过自旋的形式尝试获取锁，不会阻塞，提高性能。

  • 重量级锁：
    当锁为轻量级锁的时候，另一个线程虽然是自旋，但自旋不会一直持续下去，当自旋一定次数的时候，还没有获取到锁，就会进入阻塞，该锁膨胀为重量级锁。重量级锁会让其他申请的线程进入阻塞，性能降低。
